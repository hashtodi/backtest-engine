# Lessons

- **Expiry-day contracts (2026-06-11):** For option-buying engines, on the weekly expiry day itself trade the NEXT weekly expiry (parquet `expiry_code == 2`), not the expiring contract — it has no decent time value. `engine/data_loader.load_data(..., "weekly")` only keeps code 1, so engines needing the roll must load WEEK codes 1+2 themselves and pick per day (reference: `engine/dema_mtf_vwap_backtest.py`).

- **tz-aware `.values` silently shifts to UTC (2026-06-24):** `pd.to_datetime(iso_with_+0530).values` (or `.to_numpy()`) converts a tz-aware series to **UTC**, shifting every NIFTY bar by −5:30 (09:15 → 03:45). When that column is later used for `between_time("09:15","15:29")` / `resample(origin="start_day", offset="9h15min")`, the anchoring is wrong — it silently keeps only the IST 14:45–15:29 tail and resamples garbage, with **no error**. This bug made the triple-ST engine's *anchored* MTF SuperTrend completely wrong (entries ~50% off vs the TradingView Pine export); rolling mode was unaffected (row-order only). Fix: parse the tz-less prefix — `pd.to_datetime(s.str.slice(0,19))` — to keep IST wall-clock. Always verify a `dt` column's time-of-day matches the source `HH:MM` before resampling. Verifying engine entries against a TradingView CSV export (bar OHLC + EMA + entry-marker columns) is the fastest way to catch this.

- **MTF higher-timeframe no-look-ahead (2026-06-24):** Bars in the parquet are START-stamped (a row labeled `T` covers `[T, T+1min)`; first daily bar is `09:15`). When attaching a higher-timeframe (e.g. 5m/10m) value to a 1-min bar, replicate TradingView `request.security(lookahead_off)`: an HTF bar with start `s` and `tf` minutes is usable at 1-min bar `T` **iff `s + tf <= T`** — i.e. it only becomes visible on the 1-min bar that *opens at/after* the HTF bar closes. Concretely, at the 1-min bar `12:35` the freshly-completed 5m bar is `12:30–12:34` and 10m is `12:25–12:34` (NOT the bars starting at 12:35, which are still forming). Implement via `avail.index = htf_start + tf; reindex(onemin_start, method="ffill")`. Reference: `engine/triple_st_ema_spread_backtest.py::attach_last_completed`.

- **Quote the saved config, not the engine default (2026-06-25):** When stating a strategy's live parameters, read the `saved_strategies/<name>.json` it actually runs with — NOT the `DayContext`/dataclass defaults in the engine. They diverge: e.g. `engine/oi_wall_backtest.py` defaults `tp_inr=1000`, but `saved_strategies/oi_wall.json` overrides to `1200` (and `wall_breach_enabled=false`). I quoted 1000 from the engine and was corrected. `parse_config` is the source of truth for which keys the JSON supplies and their fallbacks; cross-check the JSON before answering "what the strategy does today."

- **OI Wall profit-lock ("trailing") SL (2026-06-25):** `engine/oi_wall_backtest.py` supports a sticky one-time profit-lock step (not a ratchet): once live spread P&L first reaches `trail_arm_inr * lots` it arms, after which the SL is re-pinned to `trail_lock_inr * lots` (a profit level) for the rest of the day, reported as exit reason **`TSL`**. `trail_arm_inr=0` disables it. Arming + lock test are evaluated on the minute **CLOSE** (consistent with SL/TP); the exit fills at **T+1 OPEN** — so a realized `TSL` is detection-based, not a guaranteed lock (a next-bar gap can fill below it). Verified no look-ahead by independent minute-by-minute reconstruction vs the engine on real data (2025-01-30 PE: armed 10:43, TSL signal 11:01, filled 11:02, pnl exact match). Tunable in `ui/oi_wall_backtest_runner.py`.

- **Workflow `args` does NOT scope the run; confirm fan-out scope before launching (2026-06-29):** Running the quarterly re-rating filter over 12 stocks, two research agents died on a transient "Connection closed mid-response". I tried to re-run only those two by adding an `args`-based filter (`const SEL = Array.isArray(args)&&args.length ? STOCKS.filter(...) : STOCKS`) and invoking `Workflow({scriptPath, args:["POONAWALLA","RRKABEL"]})`. The `args` global did **not** reach the script — `SEL` fell back to all 12 and it silently re-spawned research for **every** stock. Two takeaways: (1) don't rely on `args` to scope a workflow subset — **hardcode the subset list in the script**, and after launching, verify what it actually spawned (`ls agent-*.jsonl` in the run's transcript dir / `grep STOCK:`); (2) **Harsh watches `/workflows` and objects hard to runaway/duplicate fan-out** — after a partial failure, prefer the minimal explicit recovery (a tiny bounded set of direct `Agent` calls, or inline work in the main thread) over relaunching the whole job, and confirm scope/cost with him first. He stopped the duplicate run, then killed even the bounded 2-agent retry, and chose to finalize the report with the 10 good stocks rather than keep spawning. The 10 completed results survive on disk at the workflow's `.output` (`result` key) — a partial workflow is not wasted.
